# System Design -- Tony's Playbook

## Design Philosophy
Build things that work, explain how they work, make them easy to use. The best solution is the one that actually gets adopted.

## When Building a System or Workflow
1. **Clarify the problem** -- what exactly are we solving? Who uses it? What's the current pain?
2. **Check for existing solutions** -- don't build what already exists
3. **Design the simplest version that solves the problem** -- resist over-engineering
4. **Prototype fast** -- working demo > perfect spec
5. **Iterate based on actual usage** -- not hypothetical requirements

## Technical Decision Framework
- **Buy vs Build**: If a tool exists that does 80% of what we need, use it. Don't build from scratch for the last 20%.
- **Automation ROI**: Only automate if it saves more time than it costs to build and maintain.
- **Maintainability**: Someone else (or future Harman) needs to understand this. Document decisions, not just code.

## Education-Specific Systems
- Student data workflows (MOE systems, school admin, CCA tracking)
- Timetabling constraints and optimization
- Communication flows (parents, staff, cluster, MOE)
- Event planning systems (Awards Day, graduation, open house)
- Portfolio and documentation systems (DSA, CCA achievements)

## Presentation Design
When building slide decks:
- One idea per slide. If it needs two, it's two slides.
- Visual hierarchy: title > key visual > supporting text
- Data slides: chart + one sentence takeaway. Never a wall of numbers.
- Use Harman's color preferences and school branding
- Build in natural pause points for audience engagement
